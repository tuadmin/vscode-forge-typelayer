## Exploration: Watermark and Shortcircuit Emitting

### Current State
Actualmente el `runMinimalValidation` frena la emisión si el archivo está 100% en blanco (`!text.trim()`). Sin embargo, si un archivo tiene espacios o comentarios (ej. `// TODO: hacer algo`), TypeScript se ejecuta y graba un archivo de salida "vacío" o inútil, desgastando el SSD y perdiendo CPU/RAM.
Por otro lado, los artefactos públicos (`.js` y `.d.ts`) se generan sin advertencias. Si un desarrollador usa "Go to definition" en su editor, puede ser llevado al `.d.ts` generado en lugar del código fuente privado (`.mts`), editarlo a mano, y perder sus cambios en la siguiente autoría porque Forge TypeLayer lo sobreescribirá.

### Affected Areas
- `src/core.js` -> `runMinimalValidation` (Acá se debería expandir la validación de qué es "vacío").
- `src/extension.js` -> Módulo donde se emite el archivo mediante API fallback o CLI externo.
- `package.json` -> Para exponer la opción de configuración opcional del watermark a los usuarios de la extensión.

### Approaches

#### 1. Regex de Comentarios (Shortcircuit)
Para determinar si vale la pena disparar el compilador, podemos descartar temporalmente los comentarios estilo JS (`/* ... */` y `// ...`) y ver si queda algún caracter. Esto ignora archivos basura pero permite exportaciones de tipos puros.
- Pros: Cero overhead, detiene a TS antes de que spawnee un proceso o haga I/O.
- Cons: Una regex simple tal vez filtre `//` dentro de un string de código (ej: `"string // no es comment"`). Pero un string válido nunca estaría solo en el archivo, así que `"".trim()` daría string, por ende compila.

#### 2. Watermark / Leyenda Autogenerada
Añadir una marca de agua. Como existen compiladores CLI (que escriben al disco directo) y el API en memoria, tenemos diferencias.
- En la API: Pre-pend directo en el string de memoria antes del `fs.writeFileSync`.
- En el CLI: Tenemos que inyectar leyendo el archivo generado y volviéndolo a escribir. (Costoso para el SSD, pero el flag es opcional).
- Alternativa óptima: ¿Y si inyectamos una string de `autogenerated` al principio sólo si la configuración está activa? Dado que los `.js/.ts` carecen de SourceMaps activos por defecto, un shift de línea no rompe un sourcemap.

### Recommendation
Implementar la regex inteligente en `runMinimalValidation` y usarla temprano en `emitEntry`.
Implementar y exponer la configuración `forgeTypeLayer.addWatermark` (booleano, por defecto `true` para la mejor DX, pero controlable). Si está activo, tras compilar, revisar los archivos finales e insertar:
`/** @generated - Forge TypeLayer - DO NOT EDIT DIRECTLY */`

### Risks
- Re-escritura en archivos CLI usa un ciclo extra de disco (+1 read +1 write). Para mitigarlo, podemos hacer que el fallback lo maneje nativo y el CLI haga append, pero agregar un header es vital y justifica el costo.
- Desfasaje de SourceMaps: No generamos sourcemaps en el scope de este proyecto actual, por ende un shift de línea en la cabecera es seguro.
